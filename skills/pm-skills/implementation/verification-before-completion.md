# Verification Before Completion — 验证先行

在声称工作完成之前没有验证，是不诚实，不是效率。

**核心原则：证据先行，永远如此。**

**违反这条规则的字面意思就是违反这条规则的精神。**

## 铁律

```
没有新鲜验证证据 = 不能声称完成
```

如果你在本次消息中没有运行过验证命令，你就不能声称它通过了。

## 门控函数

```
在声称任何状态或表达满意之前：

1. 识别：什么命令能证明这个声明？
2. 运行：执行完整命令（新鲜的，完整的）
3. 读取：完整输出，检查退出码，计数失败数
4. 验证：输出是否确认了声明？
   - 如果否：用证据陈述实际状态
   - 如果是：用证据陈述声明
5. 只有然后：才能做出声明

跳过任何步骤 = 不在验证，在撒谎
```

## 常见失败模式

| 声明 | 需要的证据 | 不够的 |
|------|----------|--------|
| 测试通过 | 测试命令输出：0失败 | 上次运行、"应该通过" |
| Linter清洁 | Linter输出：0错误 | 部分检查、推断 |
| 构建成功 | 构建命令：exit 0 | Linter通过、日志看起来好 |
| Bug修复 | 原始症状测试：通过 | 代码改了，假设修复了 |
| 需求满足 | 逐条对照清单 | 测试通过 |

## 红旗 — 停止

- 使用"应该"、"大概"、"看起来"
- 在验证之前表达满意（"好了！","完美！","完成！"等）
- 准备提交/推送/PR但没有验证
- 相信agent成功报告
- 依赖部分验证
- 想着"就这一次"
- 累了想让工作结束
- **任何暗示成功但没运行验证的措辞**

## 自我辩解预防

| 借口 | 真相 |
|------|------|
| "应该能工作了" | 运行验证命令 |
| "我很确定" | 确定性 ≠ 证据 |
| "就这一次" | 没有例外 |
| "Linter通过了" | Linter ≠ 编译器 |
| "Agent说成功了" | 独立验证 |
| "我累了" | 累 ≠ 合理 |
| "部分检查够了" | 部分证明不了什么 |
| "换了个说法所以规则不适用" | 精神大于字面 |

## 关键模式

**测试：**
```
✅ [运行测试命令] [看到：34/34 pass] "所有测试通过"
❌ "应该通过了" / "看起来正确"
```

**构建：**
```
✅ [运行构建] [看到：exit 0] "构建通过"
❌ "Linter通过了"（Linter不检查编译）
```

**需求：**
```
✅ 重读计划 → 创建清单 → 逐条验证 → 报告缺口或完成
❌ "测试通过了，阶段完成"
```

**Agent委托：**
```
✅ Agent报告成功 → 检查VCS diff → 验证变更 → 报告实际状态
❌ 相信Agent报告
```

## 何时适用

**永远在以下之前：**
- 任何"完成/成功"类声明
- 任何满意表达
- 任何关于工作状态的正向陈述
- 提交、PR创建、任务完成
- 转到下一个任务
- 委托给agents

**规则适用于：**
- 精确措辞
- 释义和同义词
- 成功的暗示
- **任何暗示完成/正确性的沟通**

## 与 pm-knowledge 的衔接

验证完成后：
- 验证结果写入 `.pm-wiki/log.md`
- 发现的缺口写入 `.pm-wiki/constraints/` 或 `.pm-wiki/synthesis/`

## 底线

**验证没有捷径。**

运行命令。读取输出。然后声称结果。

这条规则没有例外。
