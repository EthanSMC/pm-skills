# 实施阶段

按照规划阶段产出的实施计划，逐步执行代码变更。

## 前置条件

- writing-plans skill 已完成，实施计划已创建
- 每个任务有明确的验收标准

## 执行原则

1. **按顺序执行** — 遵循规划中的任务顺序，不跳步
2. **一个任务一个提交** — 每完成一个任务，提交一次
3. **边写边测** — 每个任务完成后立即验证
4. **保持简洁** — 只实现计划中定义的内容，不添加额外功能

## 每个任务的执行流程

```
1. 标记任务为 in_progress
2. 阅读相关代码，理解上下文
3. 实施变更
4. 运行测试（如有）
5. 验证验收标准
6. 标记任务为 completed
7. 提交变更（如用户要求）
```

## 代码规范

- 遵循项目现有的代码风格和模式
- 不添加不必要的注释、日志或调试代码
- 不引入计划外的依赖
- 不做计划外的重构
- 错误处理只在系统边界做（用户输入、外部 API）

## 遇到问题时的处理

- **计划有误** → 暂停，向用户说明问题，讨论调整方案
- **发现新需求** → 记录到 pm-knowledge 的 requirements/ 目录，不在本次实施中处理
- **技术阻碍** → 尝试替代方案，如果不行则暂停并说明

## 与 pm-knowledge 的衔接

实施过程中发现的新信息：
- 新的约束 → 写入 `constraints/`
- 新的决策 → 写入 `decisions/`
- 新的假设 → 写入 `constraints/assumptions.md`

## 完成后

- 所有任务标记为 completed
- 运行完整测试套件
- 准备进入 review 阶段
