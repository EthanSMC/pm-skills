# 规划阶段

将 brainstorming 产出的设计文档转化为可执行的实施计划。

## 前置条件

- brainstorming skill 已完成，设计文档已通过用户审核
- 设计文档位于 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`

## 输入

读取设计文档，理解：
- 架构决策
- 组件划分
- 数据流
- 错误处理策略
- 测试要求

## 输出

实施计划，包含：
- 任务分解列表（每个任务可独立完成）
- 实施顺序（依赖关系驱动）
- 每个任务的验收标准
- 风险点和注意事项

## 执行步骤

1. **读取设计文档** — 从 specs 目录读取已审核的设计
2. **识别实施单元** — 将设计拆分为可独立实施的小单元
   - 每个单元应能独立理解和测试
   - 优先拆分基础设施层（数据模型、核心接口），再拆分功能层
3. **确定依赖关系** — 哪些单元必须先完成才能开始其他单元
4. **规划实施顺序** — 按依赖关系排序，同时考虑：
   - 风险高的先做（尽早发现问题）
   - 核心路径优先（保证主线可用）
   - 独立单元可并行
5. **定义验收标准** — 每个任务完成后应达到什么状态
6. **写入计划文档** — 保存到 `docs/superpowers/plans/YYYY-MM-DD-<topic>-plan.md`

## 任务分解原则

- **一个任务一个焦点** — 不要把多个独立变更混在一个任务里
- **可验证** — 每个任务完成后可以验证（测试、手动检查、状态确认）
- **可回滚** — 如果一个任务失败，不应该影响已完成的任务
- **大小适中** — 太大则难以跟踪进度，太小则管理开销过高

## 与 pm-knowledge 的衔接

规划完成后：
- 将实施计划的关键决策写入项目知识库 `decisions/` 目录
- 更新 `constraints/` 中新发现的依赖和约束
- 在 `log.md` 中记录规划活动

## 使用 TodoWrite 工具跟踪进度

为每个子任务创建 todo 条目，按顺序执行。状态流转：
- pending → in_progress → completed
