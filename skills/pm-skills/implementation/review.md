# 审查阶段

对实施结果进行全面审查，确保代码质量、安全性和功能正确性。

## 前置条件

- execute skill 已完成，所有代码变更已提交

## 审查维度

### 1. 设计一致性
- [ ] 实施结果是否符合设计文档的意图
- [ ] 是否有遗漏的功能点
- [ ] 是否有计划外的功能（YAGNI 检查）
- 读取 `docs/pm/specs/` 中的设计文档进行对照

### 2. 代码质量
- [ ] 代码是否简洁，没有冗余逻辑
- [ ] 命名是否清晰，能表达意图
- [ ] 函数/方法是否职责单一
- [ ] 是否有不必要的抽象

### 3. 安全性
- [ ] 用户输入是否经过验证
- [ ] 是否有注入漏洞（SQL、XSS、命令注入）
- [ ] 敏感信息是否妥善处理（不硬编码、不暴露）
- [ ] API 调用是否有适当的权限检查

### 4. 测试
- [ ] 关键路径是否有测试覆盖
- [ ] 边界条件是否测试
- [ ] 错误场景是否测试
- [ ] 测试是否可靠（不依赖外部状态）

### 5. 可维护性
- [ ] 新代码是否符合项目现有模式
- [ ] CLAUDE.md 是否需要更新
- [ ] 是否有需要记录的架构决策

## 审查流程

1. **读取设计文档** — 回顾设计意图
2. **逐文件审查** — 检查每个变更文件
3. **运行测试** — 确保所有测试通过
4. **输出审查报告** — 列出发现的问题和建议
5. **修复或记录** — 严重问题立即修复，次要问题记录到知识库

## 审查报告格式

```
## 审查报告

**审查范围**: <变更的文件列表>
**对照设计**: <设计文档路径>

### 通过项
- ✅ ...

### 问题项
- ❌ [严重] <问题描述> → <修复建议>
- ⚠️ [建议] <问题描述> → <改进建议>
```

## 审查通过标准

- 0 个严重问题
- 设计一致性 100%
- 关键路径测试覆盖
- 无安全漏洞
